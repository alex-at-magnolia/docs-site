FAQs

The following list contains answers to the most frequently occurring questions related to Magnolia.

How can I add a new language to my site?

Adding a language to a site involves defining a new locale in the configuration of your site. For further details see Define locales in site.

How can I export content from one environment, for example production,

to another (test)?

You can use the capabilities of the Backup module. It allows making a manual backup of content and configuration, which you can then restore on the new instance. For further details see Making a manual backup and Restoring a backup. See also the https://wiki.magnolia-cms.com/display/WIKI/== How+to+replicate+a+public+instance[== How to replicate a public instance] page on the Magnolia Community Wiki site.

In the context of Magnolia Cloud, you can copy the content in one environment to another for testing purposes. For more details on this please refer to the Copying content for testing page. See also Backing up and restoring in Magnolia Cloud.

How can I know if some content is served from cache or not?

The first and a high-level indicator of some content being served from the cache is provided in the Cache Tools app, which shows == how many elements are in the cache. To obtain a more detailed overview of cached objects you can use the Cache Browser app. For more details see Cache Tools app and Cache Browser app.

How can I remove the site name from the URL (EE and EE pro)?

Let’s say your site name is my-site-one and you want to access it via your own domain name my-domain.com. You probably don’t want to have the site’s name in the URL like my-domain.com/my-site-one. To avoid this, disable the uri-starts-with-sitename rule in the public context in the definition of your site. For further details see Managing cross-site access.

I cannot make POST request to Delivery endpoints, why?

The Delivery endpoints have been designed for tasks that need to obtain JCR data and, therefore, these endpoints only allow GET calls. If you need to manipulate JCR nodes or perform duties within the system, use one of the following endpoints: Nodes, Properties, or Commands.

I need a group of users to access only some specific apps, how can I set it up in Configuration?

You can control access to apps by listing a user role under the permissions node in the configuration of an app, see App permissions. The permission to use an app can also be specified under the permissions node in the app descriptor when developing the app. This allows you to provision the app to certain users in your organization. For further details see App descriptor: Permissions and Groups .

I need a user/role/group to edit only a tree of pages, how can I avoid them to access to the rest of the pages?

Use the Access control lists tab of the Security app. In an Access control list you can specify the type of access for any node or subnode in any workspace, which means also for any page tree in the website workspace. For further details see Roles and access control lists and Group and role strategies.

I would like to change an existing definition, how can I do it?

To change an existing definition, first find the definition using the Definitions app. Browse to the definition, highlight it and depending on its type you’ll be offered either the Show in Resources app or the Show in Configuration app action in the Action bar. Click the action and in the target location edit the definition.

  • In the Resources app (=Resource Files app) you can choose to hotfix the definition in the app or edit it on the file system.

  • In the Configuration app you can edit the definition in the JCR.

For more details see Resource Files app and Configuration app.

You can also change an existing definition by decorating it. For more details see Definition decoration.

Is it possible to delete one workspace from cache?

If you need to exclude content of a workspace from being cached, register the name of that workspace under the /modules/cache/config/contentCaching/defaultPageCache/flushPolicy/policies/flushAll/excludedWorkspaces node. By default a workspace cache is flushed whenever some new or modified content is detected in it. For further details see Flush policy.

What are version handlers and when should I use one?

A version handler is a Java class which is used to update the JCR during a module upgrade. They should be used to change configuration which may have been bootstrapped during the initial installation or previous upgrade.

For creating a version handler, see Configuring a custom bundle: Create a version handler.

What can I decorate?

It is possible to change definitions such as app descriptors, dialogs, field types, message views, templates, media editors and renderer definitions. Any definition bound to the Registry can be decorated. Registered items can be seen in the Definitions app .

Decorated definitions can originate from any source, including the JCR, YAML files or even executable code like Blossom. However, decorators themselves can only be defined via YAML. YAML is currently the only way to decorate already existing definitions. See what items can be decorated.

For more details see Definition decoration.

What can I define in light modules?

For a list of items that you can define in a light module please see Light modules: What items can be defined in a light module?

When I have complex dialog definions with Multivalue fields and

composite fields I get an error when I try to save it, why?

The root cause is probably the NoOpCompositeTransformer preventing the composite field from saving the field values. The solution is to use the MultiValueSubChildrenNodePropertiesTransformer for the parent multivalue field. For further information see Multivalue field: Multivalue containing a composite field and MGNLUI-2591.

Why should I use a prototype?

Prototype is an optional templating mechanism that offers a number of benefits. It ensures uniformity across templates, avoids repeating and duplicating configuration and creates similar templates quickly.

If you have a small number of templates that are very different from each other, using a prototype is probably not necessary. Creating a page template prototype makes more sense for a very high number of page templates that reuse a configuration. With a template prototype the template configuration is more efficient as you only need to apply a configuration change once in the template prototype. For more details see Template prototype.

Why should I use Image variations?

Variations are an effective alternative to resizing images with CSS. The main benefits of using image variations are the following:

  • Image size is reduced and less data transferred.

  • Editors can upload images of any size which will automatically be resized to fit the template.

  • Image sizes are uniform throughout the site.

For more information see Theme: Image variations.
